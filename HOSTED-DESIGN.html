@mordn/chat-widget — Hosted Backend Plan

Evidence-backed infra design · grounded in how OpenAI, Cursor & Chatbase actually scale · drafted June 2026 · light theme

The reassuring headline. We researched how products serving hundreds of millions are built. The verdict: you do not need anything exotic. OpenAI runs ChatGPT for 800M users on a single-primary Postgres + read replicas — no sharding. The architecture we already designed is the right one. The thing that determines scale is the data topology (which the real products agree on), not the compute platform (which they all differ on, and all work).

1 · What the real products do — evidence

ProductScaleWhat they run
OpenAI / ChatGPT ~800M users Single-primary Postgres + ~50 read replicas across regions. No sharding (deliberately — "too complex for a read-heavy workload"). PgBouncer (50ms→5ms). Redis cache with stampede-lock. Replicas co-located with app per region → low double-digit-ms p99 globally.
Cursor 1M+ QPS, billions/day AWS backend (EC2, managed by Terraform). Cloudflare as a reverse proxy in front (TLS, DDoS, perf). A TypeScript monolith (Rust hot paths) — not microservices. Latency POPs in Tokyo + London on a US core.
Chatbase $4M ARR, 500K MAU Vercel + Next.js + AI SDK. Optimised for iteration speed over infra cost — right call at their stage. (Proof the platform is flexible.)
The pattern — three platforms, one data architecture: single-primary Postgres + many regional read replicas + PgBouncer + a Redis cache + object storage for blobs; co-locate compute with replicas for global latency; don't shard until forced. Cursor=AWS, OpenAI=Azure/K8s, Chatbase=Vercel — all scale, because they all got the data layer right.

3 · Global low-latency — the proven way

"Fast everywhere" is a data problem, not a compute one. OpenAI's answer (and now ours):

  • Read replicas in each region, co-located with API compute → reads served locally (history loads, conversation lists).
  • Writes go to the primary (a message insert tolerates one cross-region hop; reads dominate chat).
  • Cloudflare edge in front routes users to the nearest region and absorbs TLS/DDoS.
  • Redis per region caches hot conversations so most reads never touch Postgres at all.

This is regional read-replica co-location — exactly OpenAI's "double-digit-ms p99 globally" technique. Not a magic platform; a data topology.

4 · Two-level identity & tenancy

Tenant

"Whose app?" — from the API key. Shared cluster, tenant_id on every row. Isolates customers.

End user

"Which of their users?" — the app passes its server-verified user id (existing getUserId). Isolates users within a tenant.

Both filtered on every query. tenant_id threaded from day one (even on one shared cluster) is the keystone that lets an enterprise tenant later move to a dedicated DB without a rewrite. OpenAI's "noisy-neighbor" lesson: route low/high-priority tenants to separate replica pools as you grow.

Each request: authenticate tenant (key) → read user (header) → createDrizzleChatStore(userId) scoped to tenant, against the regional replica (reads) or primary (writes) → JSON. The package-side createHostedChatStore({ apiKey }) is a thin HTTP client over the same ChatStore interface (≈ a day's work once the API exists).
